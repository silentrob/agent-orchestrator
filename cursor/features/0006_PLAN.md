## SCOPE FENCE

### IN-SCOPE

- **Trust Vector persistence:** represent **satisfied gates** (subset of `TrustGateKind`) so `requireIssueLifecycleGates` can allow **executor**-phase spawns when policy is met, not only block with the current stub message.
- **Between-phase gates:** minimum viable transitions **reproducer → plan → execute → validate → done** with explicit **preconditions** (machine + human) drawn from `TRUST_GATE_KINDS` and `issue-lifecycle-trust-vector.md` §3–4.
- **Thin prompt layers:** replace or deepen **placeholder** text in `buildIssueWorkflowPhaseLayer` for `execute`, `validate`, and `reproducer` so agents receive Trust-aligned instructions comparable in rigor to `buildPlannerArtifactLayer` (still one agent stack; no new plugins).
- **Plan artifact gate:** optional **read** of `.ao/plan.md` YAML frontmatter (path from metadata `planArtifactRelPath` or default `.ao/plan.md`) to satisfy **`human_plan_approval`** / **`artifact_plan_present`** when `requireIssueLifecycleGates` or future policy requests it.
- **Visibility:** surface **next missing gate** or **satisfied gates** on CLI (`ao status`) and/or web session detail when metadata is present (minimal UI; existing Tailwind patterns).
- **Tests + typecheck** for touched core paths; web tests if UI changes.

### OUT-OF-SCOPE

- Full **policy engine** (`docs/specs/development-policy-engine.md` scope) — no arbitrary rule DSL in this plan.
- **Tracker plugins** pushing gate state to GitHub/Linear as source of truth (may be a later optional mirror).
- **New npm agent packages** per role — roles remain `WorkerRole` + prompts + metadata.
- **Artifact manifest / `ao artifact publish`** — not required for gate MVP.
- **0003-style** HTTP approve routes and snapshot writers unless explicitly revived in a separate plan.

## NO-INVENTION RULE

Any **new** metadata key names, file paths, or public functions appear only in **Delta Proposals** with exact signatures.

## Context Summary

**Plan 0005** shipped **issue workflow phase** on sessions, **phase-aware `buildPrompt`**, **thin** executor/validator/reproducer layers, dashboard/CLI **phase** visibility, and a **spawn guard** (`requireIssueLifecycleGates`) that **rejects executor-phase spawns** because **Trust Vector satisfaction is not yet persisted** (`session-manager.ts` guard). The **thinnest** parts are: (1) **no durable gate vector**, (2) **placeholders** for non-planner phases, (3) **no automated bridge** from `.ao/plan.md` frontmatter to “gate satisfied,” (4) operators cannot see **what gate blocks** the next step.

This plan (**0006**) specifies engineering work to **flush out** those gaps with **emphasis on trust gates between steps**, staying aligned with `issue-lifecycle-trust-vector.md` and the persistence sketch in `issue-lifecycle-state-persistence.md`.

## Data Layer (Phase 1)

**Minimum:** extend **session metadata** (`updateMetadata` / flat key=value) with one of:

- **Option A:** namespaced keys, e.g. `trustGateHumanPlanApproval=satisfied` / `trustGateCiPassing=pending` (string enum per gate), or
- **Option B:** single key `trustGateVector` = compact JSON string (must validate size; document max length).

**Decision (approved): Option A** — MVP slice gates: `artifact_plan_present`, `human_plan_approval`, `ci_passing` as separate metadata keys derived from `TrustGateKind` (see implementation tasks). Option B deferred.

**Canonical issue state (optional follow-on in same plan if small):** issue-index file under project data dir per `issue-lifecycle-state-persistence.md` Candidate 2 — **only** if session churn makes metadata loss unacceptable; otherwise defer.

## Implementation

### a) Core — gate model + spawn evaluation

1. Add **stable string constants** for metadata keys (new file or `issue-lifecycle-types.ts`) — **Delta Proposal §1**.
2. Implement **`readGateState` / `mergeGateSatisfaction`** helpers** (pure functions + file read for `.ao/plan.md` when workspace path known) — **Delta Proposal §2\*\*.
3. Replace the **boolean-only** branch in `createSessionManager` → `spawn` when `project.requireIssueLifecycleGates`: instead of rejecting all executor spawns, **evaluate** required gates for transition into `execute` (e.g. plan file exists + frontmatter `status: approved` when `requires_approval` was true). If not satisfied, **throw** with a message listing **missing gates** (operator-actionable).
4. Wire **lifecycle** or **post-PR** hooks (minimal): when SCM reports CI green, set `ci_passing` satisfied on the session metadata for that issue worker (exact hook point TBD in implementation; may reuse existing lifecycle polling in `lifecycle-manager.ts`).

### b) Prompts — deepen `artifact-layers-by-role.ts`

1. Expand **`buildExecutorPhasePlaceholder`** (rename to substantive `buildExecutorArtifactLayer` if size warrants) with: Trust checklist, PR body expectations (already partially in base prompt), **preconditions** referencing gates.
2. Expand **validator** and **reproducer** sections to match **role-typed-artifacts** intent (verification evidence, repro minimality) — still markdown strings only.

### c) API / visibility

1. **`ao status`:** optional column or dim suffix: **next gate** or **gates: a,b** when metadata keys present (`packages/cli/src/commands/status.ts`).
2. **Web:** extend `sessionToDashboard` / `SessionDetail` to show gate summary when keys present (`packages/web/src/lib/serialize.ts`, components).

### d) Documentation

1. Short cross-link from `docs/specs/issue-lifecycle-state-persistence.md` Recommendation to implemented keys (when merged).
2. No large doc rewrites — pointer updates only.

### Delta Proposal §2 — Plan frontmatter probe

```typescript
// e.g. packages/core/src/plan-artifact-gates.ts (new)
export interface PlanFrontmatterProbeResult {
  path: string;
  found: boolean;
  status?: string;
  requiresApproval?: boolean;
}
export function probePlanArtifact(
  workspacePath: string,
  relPath?: string,
): PlanFrontmatterProbeResult;
```

(Use a **small** YAML frontmatter parser or line-based `---` slice consistent with existing tests in `spawn.test.ts` that write `---\nstatus: draft\n---`.)

### Risks

- **False positives:** approving executor spawn from file alone — mitigate by requiring **issueId match** `planArtifactIssue` in metadata when present.
- **Concurrency:** two spawns same issue — document **canonical session** rule (first active worker for issue) or gate only on explicit CLI spawn.
