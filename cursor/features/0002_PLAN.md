## Context Summary

This plan covers a **proof-of-concept** for the **Planner** worker role described in `docs/specs/role-typed-artifacts.md`: the **orchestrator** can trigger a dedicated planner session (same agent stack as today, different instructions), the planner produces a **plan artifact** as a concrete file, and the **web dashboard** can **read and display** that content for humans. Full session-artifact manifest tooling (`ao artifact publish`, role-scoped kind enforcement, v5 storage) remains **out of scope** for this POC.

The plan file is treated as a **first-class object logically attached to the issue** (via session `issue` metadata + stable path under the issue’s worktree): future **policy** (e.g. “no executor until `status: approved`”) can key off parsed frontmatter and/or session metadata without adopting the full v5 manifest in this phase.

**Practical POC convention:** optional **YAML frontmatter** in `.ao/plan.md`, aligned with `cursor/features/*_PLAN.md` style (`status`, `requires_approval`, etc.). **`requires_approval` defaults to `false` for the POC** so gates are opt-in per plan, not blocking by default. Frontmatter is **authoritative for display and future automation hints**, not a security boundary until validation and a single source of truth are defined.

**Respawn / crash continuity:** If a planner session **OOMs or crashes**, a replacement session on the **same issue** should **reuse the same worktree path when workspace reuse applies**, or otherwise **rediscover** the plan via conventional location + metadata. Prompt text for planner respawn should instruct **read → update** the existing file rather than blindly overwriting, so the artifact remains continuity for humans and orchestrator context (exact restore wiring stays within existing `SessionManager.restore` / spawn behavior — see Implementation).

## Scope Fence

### IN-SCOPE

- **Spawn path:** Ability to create a worker session that is tagged as a planner (persisted in session metadata) and receives **planner-specific prompt text** (decomposition / implementation PR rules relaxed or replaced in instructions only — no change to lifecycle state machine).
- **Orchestrator operability:** Documented (and optionally prompt-assisted) **CLI workflow** so the orchestrator agent uses existing `ao spawn` (or extended flags) to start a planner session for an issue.
- **Artifact for POC:** A **single conventional file** under the session worktree (e.g. `.ao/plan.md`) written by the planner agent per instructions; optional metadata key pointing at the resolved path for discovery.
- **Plan frontmatter (POC convention):** Optional YAML block at top of `.ao/plan.md` (see **Plan artifact convention** below) including `status`, `requires_approval` (default **`false`** if omitted), and optional identifiers — enables dashboard badges and **future policy** without implementing enforcement in this POC.
- **Issue attachment (logical):** Session metadata already stores **`issue`**; plan path is under that issue’s worktree so the artifact is **keyed by issue + project** for listing and policy. Optional metadata key **`planArtifactIssue`** (**PROPOSED**) may duplicate issue id for sanity checks when reading from disk.
- **Respawn continuity:** Document and, where minimal, implement **planner prompt guidance** so after crash/OOM a new planner session **loads existing `.ao/plan.md`** if present and updates it (preserve frontmatter semantics). Align with existing **`SessionManager.restore`** where the same session id returns (PROPOSED: mention existing plan in restore context — Delta if new prompt injection is required).
- **Web:** A **read-only** HTTP API that returns plan **body**, **parsed frontmatter** (when present), and resolved path for a session id; minimal **Session detail UI** (content + optional status badge from frontmatter).
- **Safety:** Server-side path resolution **confined to `session.workspacePath`** (no arbitrary filesystem read).

### OUT-OF-SCOPE

- Full **v5 manifest**, `ao artifact` CLI, `validatePublish`, `producerRole` / `ArtifactKind` enforcement, and `inputArtifactIds` lineage as specified for production.
- **Automated** executor/validator handoff **blocked** on `status: approved` (policy enforcement is **future**; POC only defines fields and parsing for hooks).
- **Tracker plugin** as the primary artifact store or mirroring full plan bodies into issues (optional comment/link only, not required).
- **Telemetry**, analytics, SSE interval changes, or broad dashboard redesign.
- **New plugin slot** or separate “planner” binary; planner remains a normal **worker** session with different prompt + metadata.

## Data Layer (Phase 1)

- **Session metadata** (`updateMetadata` / key=value files): optional keys **PROPOSED** — `workerRole`, `planArtifactRelPath`, optionally `planArtifactIssue` (must match session `issue` when both set, for policy hardening later). Not yet part of the typed `SessionMetadata` interface in `types.ts`.
- **Plan file:** `.ao/plan.md` (or `planArtifactRelPath` relative to worktree) containing:
  - Optional **YAML frontmatter** between `---` delimiters (POC schema below).
  - **Markdown body** (human-readable plan).
- **No DB**; issue association is **`issue` in session metadata** + worktree scope.

## Plan artifact convention (POC)

**File:** default `.ao/plan.md` under `workspacePath` (overridable via `planArtifactRelPath`).

**Frontmatter (optional; align with feature-plan style):**

| Field               | Type / values                                                | POC default                                  | Notes                                                                  |
| ------------------- | ------------------------------------------------------------ | -------------------------------------------- | ---------------------------------------------------------------------- |
| `status`            | `draft` \| `needs_clarification` \| `approved` \| `rejected` | `draft` if frontmatter present without value | Dashboard badge; future policy reads this.                             |
| `requires_approval` | boolean                                                      | **`false`**                                  | If `true`, signals intent for a future gate; **no auto-block** in POC. |
| `feature_name`      | string                                                       | optional                                     | Short title for UI.                                                    |
| `issue_id`          | string                                                       | optional                                     | Should match tracker issue; enables cross-check with session `issue`.  |

**Planner agent instructions (via `buildPlannerArtifactLayer` or `userPrompt`):** Must state that the planner **creates or updates** this file, **preserves valid YAML frontmatter** when editing, sets `status` appropriately, and uses **`requires_approval: false`** unless the user/orchestrator explicitly wants a review gate.

**Trust model:** POC treats frontmatter as **advisory** (agent-written). Production policy would validate + optionally require human/dashboard-signed metadata (Future Work).

## Implementation

### A) Data / service logic (core)

1. **Extend `SessionSpawnConfig`** (see Delta Proposal) so `createSessionManager` → `spawn` can accept optional **`workerRole`** (string union **PROPOSED**) and rely on existing **`prompt?: string`** for planner instructions.
2. In `packages/core/src/session-manager.ts` inside `spawn`, after successful `writeMetadata` / post-launch metadata updates, call **`updateMetadata(sessionsDir, sessionId, updates)`** when planner-related fields are set (**PROPOSED** keys): `workerRole`, optional default **`planArtifactRelPath=.ao/plan.md`** for `workerRole === "planner"`, and optional **`planArtifactIssue`** mirroring `spawnConfig.issueId` for future policy checks.
3. **Compose prompt for planner:** Either:
   - **(POC-minimal)** Pass through existing `buildPrompt({ … userPrompt: spawnConfig.prompt })` and require caller to pass full planner instructions in `prompt`, or
   - **(POC-aligned with spec)** After Delta: import **`buildPlannerArtifactLayer`** (**PROPOSED**) and concatenate its output with existing `buildPrompt` output before agent launch (exact wiring **PROPOSED** — not an existing API today). Layer text must include **`.ao/plan.md`**, **frontmatter fields** (defaults above), and **respawn behavior** (if file exists, read and update; do not discard without explicit user direction).
4. **Respawn / same-issue replan:** When spawning a planner for an issue where **`.ao/plan.md` already exists** (e.g. prior session died), **`buildPrompt` / planner layer** should inject a short **“Existing plan on disk”** section: path + instruction to open, reconcile, bump `status` if needed (**PROPOSED** — may require optional pre-spawn filesystem probe in `session-manager.spawn` or rely on agent discovery only for minimal POC).
5. **`SessionManager.restore`:** If restore reattaches to a planner session with an existing worktree, ensure metadata still exposes `workerRole` / `planArtifactRelPath` so the web plan API keeps working (**PROPOSED** audit of restore path — no new API unless gap found).
6. **`generateOrchestratorPrompt`** (`generateOrchestratorPrompt(opts)`) — append a short **“Planner workflow (POC)”** subsection documenting the exact `ao spawn` invocation (issue id, flags), **plan file location**, **frontmatter meaning**, and **crash recovery** (respawn planner same issue to continue plan).

### B) API / CLI layer

1. **CLI** (`packages/cli/src/commands/spawn.ts`): Wire **optional flags** (**PROPOSED**): e.g. `--prompt <text>` and `--worker-role planner` → forward to `sessionManager.spawn({ projectId, issueId, prompt, workerRole })`. Today `spawnSession` only passes `projectId`, `issueId`, `agent` (see Reference Proof).
2. **Web** (`packages/web/src/app/api/spawn/route.ts`): Optionally extend `POST` body to accept `prompt` and `workerRole` and pass through to `sessionManager.spawn` (same fields as core config). Currently only `projectId` and `issueId` are forwarded (see Reference Proof).

### C) UI layer (web)

1. Add **`GET /api/sessions/[id]/plan`** (or equivalent name) in `packages/web/src/app/api/sessions/[id]/` that:
   - Uses **`getServices()`** → **`sessionManager.get(sessionId)`** (`SessionManager` interface).
   - If `session` missing or **`workspacePath`** null → 404.
   - Resolves candidate path: default **`<workspacePath>/.ao/plan.md`** or path from **`session.metadata["planArtifactRelPath"]`** (**PROPOSED** key), with **`resolve` + `realpath` / prefix check** so resolved path stays under workspace (**PROPOSED** helper in web `lib/` or small core util — **PROPOSED** if not present).
   - Parses optional YAML frontmatter (**PROPOSED** helper `parsePlanMarkdown(raw: string): { frontmatter: Record<string, unknown>; body: string }` — implement with a small dependency or manual `---` split; not in core today).
   - Returns JSON `{ path, body, frontmatter, issueId: session.issueId }` (shape **PROPOSED**) so UI can show **status** / **requires_approval** without re-parsing client-side.
2. **`SessionDetail`** (`packages/web/src/components/SessionDetail.tsx`): When `session.metadata["workerRole"] === "planner"` (or equivalent), **fetch** the plan endpoint and render **body** plus compact **badges** from frontmatter (`status`, `requires_approval` if true) — read-only markdown or `<pre>` (Tailwind only, no new UI libraries per project rules).

## Risks & Delta Proposals

| Item                               | Risk   | Mitigation                                                                                                        |
| ---------------------------------- | ------ | ----------------------------------------------------------------------------------------------------------------- |
| Path traversal via metadata        | High   | Strict workspace prefix check after `resolve`.                                                                    |
| Large plan files                   | Medium | Cap response size in route (e.g. first 256KB) **PROPOSED** constant.                                              |
| `writeMetadata` vs arbitrary keys  | Low    | `SessionMetadata` type omits `workerRole`; use **`updateMetadata`** for extra keys after initial `writeMetadata`. |
| Agent corrupts YAML frontmatter    | Medium | Prompt constraints + optional server-side YAML parse errors surfaced as “unparsed” in API.                        |
| Frontmatter vs tracker issue drift | Low    | Optional `planArtifactIssue` metadata check against `session.issueId`; policy layer later.                        |
