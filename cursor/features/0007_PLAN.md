## SCOPE FENCE

### IN-SCOPE

- **CLI:** `ao plan <sessionId> approve` — operator marks the **plan artifact** (`.ao/plan.md` or `planArtifactRelPath`) as **human-approved** by updating YAML frontmatter so it matches **0006** gate semantics (`status: approved` per `isHumanPlanApprovalSatisfied` in `evaluate-trust-gates.ts`).
- **CLI:** `ao plan <sessionId> send <message...>` — **quality-of-life** alias that forwards to the same path as **`ao send`** (session manager `send`) so feedback to a **planner** session is grouped under `plan` without reimplementing tmux/OpenCode delivery.
- **Web:** An **Approve** control on the planner **Plan** panel (when `requires_approval` is true and plan is not yet approved) that performs the **same** approval as the CLI (via a new HTTP handler), then refreshes plan read state.
- **Safety:** Path containment for plan file writes (mirror `resolvePlanArtifactPath` rules used by `GET /api/sessions/[id]/plan`).
- **Tests:** Core unit tests for any new write helper; CLI command tests; web tests for button + route.

### OUT-OF-SCOPE

- **Full policy engine** or arbitrary frontmatter editors (`docs/specs/development-policy-engine.md`).
- **Tracker/GitHub** mirroring of approval state (labels, comments) — optional follow-up.
- **`ao plan` subcommands** for every frontmatter state (`draft`, `rejected`, …) beyond **approve** and **send** — capture as Future Work unless folded into the same MVP with minimal extra surface.
- **Rewriting plan body** or AI-assisted revision (only structured **send** message to agent).
- **Breaking changes** to existing `ao send` / `POST /api/sessions/[id]/message` contracts.

## NO-INVENTION RULE

New **public** functions, routes, or metadata keys appear only in **Delta Proposals** with exact signatures.

## Context Summary

Plans **0005** (issue workflow phase, planner UI read-only plan) and **0006** (Trust gates; executor spawn requires **`human_plan_approval`** satisfied when `requires_approval` is true, which means **`status: approved`** in plan frontmatter per `isHumanPlanApprovalSatisfied`) are **complete**, but operators still **edit `.ao/plan.md` by hand** to approve. The user wants **`ao plan <id> approve`**, **`ao plan <id> send "…"`** (user wrote `ap`; interpret as **`ao`** CLI), and a **web Approve button** so approval is one action from CLI or dashboard.

## Implementation

### a) Core — plan frontmatter write (Delta)

1. Add a **single** well-tested helper (new module or extend `plan-artifact-gates.ts`) that:
   - Resolves absolute plan path inside `workspacePath` (same containment rules as web `resolvePlanArtifactPath` — either **move** that helper to core for reuse, or **duplicate** minimal logic with tests).
   - Reads markdown, parses first `---` … `---` block (reuse or share parser with `parsePlanFrontmatter` / `parsePlanMarkdown` patterns).
   - Writes back with **`status: approved`** and optional **`approved_at`**, **`approved_by`** (ISO8601 / string) without destroying body or unrelated keys.
2. Export from `@composio/ao-core` and reference from CLI + web API.

### b) CLI — `ao plan`

1. Add **`registerPlan(program)`** in `packages/cli/src/commands/plan.ts` (name aligns with user request; avoid collision with **`decompose`** / **`spawn`**).
2. Subcommands:
   - **`approve <session>`** — `loadConfig` → `getSessionManager` → `get(session)` → require `workspacePath` → call core approve helper → print success / errors.
   - **`send <session> [message...]`** — delegate to **`sessionManager.send(sessionId, message)`** (same as `registerSend` success path when session exists); support `-f/--file` if easy to mirror from `send.ts`.
3. Register in `packages/cli/src/index.ts` next to other `register*` calls.

### c) Web — Approve button + API

1. Add **`POST /api/sessions/[id]/plan/approve`** (or `PATCH` on `plan` — prefer **POST** + sub-resource for clarity) that:
   - Validates session id (same as `message` route).
   - Loads session; checks **planner** context (`workerRole=planner` or equivalent metadata used today for `PlannerPlanPanel` visibility).
   - Calls the **same core helper** as CLI.
   - Returns JSON `{ ok: true }` or structured error.
2. **`PlannerPlanPanel`**: when `showRequiresApproval` and status not approved, render a **primary** button **Approve plan** that `fetch`es the new route, then refetches GET plan or updates local state.

### d) Tests & docs

1. Core tests for frontmatter merge (preserve body, update keys).
2. CLI tests for `plan approve` / `plan send` (mock FS or temp workspace).
3. Web component/API tests for approve flow.
4. Short pointer in `AGENTS.md` or CLI `--help` only — **no** large doc rewrite in this plan.
