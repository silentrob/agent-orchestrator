## Context Summary

We align engineering with this **north star**: the **orchestrator** and **CLI** manage agents; **`spawn`** remains the primitive to **create** a new agent (session + runtime + workspace). **Session lifecycle** is a separate concern: operators must **see** status and **advance** issue-backed **phases** (planner → executor → validator) on an **existing** session—**policy gates** (Trust Vector) gate transitions—without treating “new spawn per phase” as the default product path. This plan captures **today’s verified surface**, what is **missing or thin**, and a **target shape** for **core + CLI** (no task list; downstream work defines tasks after approval).

## Implementation (target shape — post-approval)

**a) Core**

1. Introduce a **single entry point** for “advance this session to the next phase” (Delta §1): validate **policy gates** for the target transition, persist `issueWorkflowPhase` + `workerRole` (or derived rules), refresh trust metadata as needed, optionally **compose** the next prompt via `buildPrompt` and **`SessionManager.send`** to the running agent.
2. Reconcile **workspace path**: either **reuse** one directory per issue for all phases (workspace plugin / config change) or **define** copying/linking of `.ao/plan.md` and gate probes so executor validation reads the planner artifact (today executor spawn probes **its** `workspacePath` in `session-manager.ts`).

**b) CLI**

1. Command(s) mirroring core: e.g. `ao session advance <id> --phase execute` (exact UX in tasks doc), sharing gate evaluation with core helpers (`listMissingExecutorTrustGates` / successors for validate transitions).
2. Extend `ao status` / help text so “next gate” / “blocked transition” is **actionable** when advance is not allowed.

**c) Policy gates**

1. Today gates are **strongest** at **spawn** (executor). Extend policy so **advance** is subject to the same (or stricter) checks, so bypassing spawn cannot bypass policy.

## Risks & Delta Proposals

### Delta Proposal §1 — `SessionManager.advancePhase` (PROPOSED)

**Rationale:** No first-class transition exists on `SessionManager`; metadata is updated internally at spawn, not via a public phase-advance API.

```ts
// packages/core/src/types.ts — SessionManager (extension)
advancePhase?(
  sessionId: SessionId,
  target: { phase: IssueWorkflowPhase; workerRole?: WorkerRole },
  options?: { skipGateCheck?: boolean },
): Promise<Session>;
```

**Impact:** New surface area; tests; must interact with `buildPrompt`, `send`, `updateMetadata`, gate evaluators.

### Delta Proposal §2 — `evaluatePhaseTransitionGates` (PROPOSED)

**Rationale:** `listMissingExecutorTrustGates` is executor-spawn-specific; validate-phase transitions may need `artifact_verification_present`, `validation_signoff`, etc.

```ts
// packages/core/src/evaluate-trust-gates.ts or new module
export function listMissingTransitionGates(
  from: IssueWorkflowPhase,
  to: IssueWorkflowPhase,
  ctx: ExecutorTrustGateContext /* or generalized context */,
): TrustGateKind[];
```

**Impact:** Pure function + tests; used by `advancePhase` and CLI.
