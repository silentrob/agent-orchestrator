## Context Summary

Feature **0002** introduced `WorkerRole`, planner prompt layers, `GET /api/sessions/[id]/plan`, and dashboard visibility for `.ao/plan.md` — useful **building blocks**.

Feature **0003** doubled down on a **multi-session handoff** model: separate planner session, durable snapshots, approve/request-changes APIs, and executor spawn fed by copied plan text. That matches AO’s historical primitive (**one session ≈ one worker**), but it **does not** match the desired product: **sub-agents as phases**, **one issue**, **one worktree**, and **natural pauses** at **policy gates** (the **Trust Vector**).

This plan **supersedes 0003 as architecture direction**. Implementation may **revert 0003 commits** on `feat/0002-planner-poc` (see §7) and grow from the **0002** baseline (or a clean branch off `main` + selective cherry-picks).

**Related specs:** `docs/specs/role-typed-artifacts.md` (role contracts and artifact kinds — still valid as **semantics**), `docs/specs/plan-approval-and-orchestrator-handoff.md` (partially obsoleted for handoff **shape**; gates and trust ideas may be reinterpreted per §3).

---

## Problem

1. **Session-centric pipeline** (planner session → snapshot → new executor session) forces **duplicated context**, **easy mistakes** (`ao send` to wrong session), and **no first-class “where is this issue?”** state.
2. **Trust** (human approval, CI, policy) is **bolted on** instead of modeled as **discrete transitions** on an **issue** (or primary work unit).
3. **Roles** (planner / executor / validator) are better thought of as **phase permissions** and **artifact contracts** on **one collaboration surface**, not necessarily **three tmux sessions**.

---

## Target architecture (conceptual)

### Unit of work

- **Primary:** **Issue** (tracker id) + **one git worktree / branch** for that issue’s delivery, unless explicitly scaled out.
- **Session** (tmux/agent) is an **implementation detail**: one long-lived **issue worker** may move through **phases**, or thin **delegates** may attach later — to be decided in implementation phasing.

### Lifecycle phases (happy path)

Ordered stages with **explicit exits**; names are indicative:

| Phase        | Intent                       | Typical artifacts (role-typed)                  | Gate before next phase                                            |
| ------------ | ---------------------------- | ----------------------------------------------- | ----------------------------------------------------------------- |
| **Plan**     | Intent, scope, risks, deltas | `plan`, `decision`, `research`, `risk`          | Policy: e.g. human approval, or `requires_approval: false` + auto |
| **Execute**  | Implementation, tests, PR    | `implementation-note`, `diff-summary`, PR ref   | Policy: PR open, CI green or waived                               |
| **Validate** | Verification, sign-off       | `test-report`, `verification`, `review-summary` | Policy: validator pass, optional human                            |

**Reproducer** (debug flows) remains an optional **pre-Plan** phase per `role-typed-artifacts.md`.

### Trust Vector (policy gates)

**Trust Vector** = the set of **checks** that must pass to **advance** the issue (or to treat output as authoritative). Examples:

- **Human:** approve plan, request changes, approve merge (existing dashboard actions reinterpreted as **gates on the issue**, not necessarily on a “planner session id” only).
- **Machine:** CI status, required checks, artifact presence (`plan` exists, `verification` after execute).
- **Config:** per-project rules (who may skip gates, mandatory validator, etc.) — aligns with future **policy engine** work.

Gates are **discrete**; between gates the system may **pause** (idle, waiting_input, or explicit `awaiting_approval`).

### Relation to role-typed artifacts

- Keep **role × artifact kind** contracts and **prompt layers** as **phase-local** rules (L4.5x in `buildPrompt`), not as “each role = one spawn” by default.
- **Orchestrator prompt** should describe **issue state** and **next gate**, not only “spawn executor with `--prompt`”.

- Full **v5 artifact manifest** implementation (unless a thin slice is chosen).
- Replacing SSE cadence or full dashboard redesign.
- **Guaranteed** LLM compliance (policy + UX nudges; not formal verification).

## Delta from 0003 (explicit)

| 0003 approach                                           | 0004 direction                                                    |
| ------------------------------------------------------- | ----------------------------------------------------------------- |
| Planner **session** owns approval UX tied to session id | Gates tied to **issue** (session is secondary)                    |
| Executor = **new spawn** + snapshot/`--prompt`          | Execute phase in **same worktree** by default; spawn is scale-out |
| Trust = metadata on planner + durable file              | Trust Vector = **policy gates** on phase transitions              |
| `ao send` to planner only                               | Messaging and instructions tied to **current phase**              |
