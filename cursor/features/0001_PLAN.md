## Context Summary

Given the updated `docs/specs/recovery-context-and-learnings.md`, this plan unifies two existing directions: restart continuity (resume/fallback) and coordination-state recovery. It also adds a repository-level `LEARNINGS.md` loop for long-term agent coordination, while keeping operational runtime memory in AO project metadata storage. Prompt context will use compact markdown summaries (not raw JSON blobs).

## Scope Fence

### IN-SCOPE

- Auto-resume-first behavior for worker respawn and orchestrator respawn paths, using existing agent `getRestoreCommand()` support with fallback to context-injection.
- Coordination snapshot persistence in existing AO metadata model (minimal fields only).
- Prompt projection updates: compact `Previous Session Context`, `Current Coordination Context`, and `Team Learnings` sections.
- `docs/LEARNINGS.md` load-at-start behavior and completion reflection merge/dedupe flow.
- Optional portability command surface for export/import bundle (recovery snapshot + learnings).

### OUT-OF-SCOPE

- Full artifact platform (`ao artifact publish/list/grep/read`) and manifest indexing.
- New dashboard/UI features, telemetry expansion, or web APIs.
- Large refactors of plugin architecture or lifecycle orchestration unrelated to recovery context.
- Advanced planning subsystem expansion beyond prompt/context integration.

## Data Layer (Phase 1)

- Extend persisted session metadata keys (runtime store) via existing key-value metadata APIs:
  - `goal`, `blockedBy`, `blocks`, `coordinationSent`, `progressSummary`, `resumedFrom`, `updatedAt`.
- Add in-repo learning document:
  - `docs/LEARNINGS.md` (default; path configurable later if needed).
- No DB or schema migration required (flat metadata files + markdown file).

## Implementation

### A) Data/Service Logic

1. Add a recovery-context service module in core that:
   - reads active + archived metadata using existing metadata APIs,
   - computes continuity context for same issue/agent,
   - emits prompt-safe markdown summaries with deterministic truncation.
2. Extend session spawn behavior through `createSessionManager(...)` workflow:
   - attempt native restore command when same-issue prior metadata exists,
   - on failure/unavailable, prepend compact fallback context to prompt before launch.
3. Extend orchestrator spawn behavior through the same manager:
   - apply resume attempt when prior orchestrator metadata exists and strategy permits.
4. Add lightweight learnings service:
   - load `docs/LEARNINGS.md` if present,
   - merge completion reflection entries with dedupe rules (`Rule + Context` normalization),
   - reject low-signal entries without evidence.
5. Add portability service commands (MVP):
   - export bundle of recovery snapshot + learnings,
   - import bundle for bootstrap on another environment.

### B) API/CLI Layer

1. Add new CLI command group (MVP names, PROPOSED):
   - `ao context export`
   - `ao context import`
2. Ensure existing `ao start` and worker spawn paths automatically consume:
   - recovery context summary,
   - learnings summary (top-N relevant entries).

### C) Prompt Layer

1. Update `buildPrompt(...)` to include:
   - `## Previous Session Context`
   - `## Team Learnings` (if file exists and non-empty)
2. Update `generateOrchestratorPrompt(...)` to include:
   - `## Current Coordination Context` (recovered summary)
3. Enforce prompt formatting constraints:
   - no raw JSON injection,
   - compact bullets/table-like markdown,
   - fixed size caps with deterministic truncation.

## Risks & Delta Proposals

### Risks

- Prompt bloat from adding three new context sections.
- Metadata key sprawl can reduce maintainability if fields are not normalized/documented.
- Learnings file can become noisy without strong merge/dedupe discipline.
- Resume attempts can fail for stale/corrupt agent state; fallback chain must remain robust.

### Delta Proposals (PROPOSED)

1. **File:** `packages/core/src/recovery-context.ts`  
   **Signature:**  
   `export function buildRecoveryPromptSections(input: { projectPath: string; projectId: string; sessionId?: string; issueId?: string; maxChars?: number }): { previousSession?: string; coordination?: string };`  
   **Rationale:** centralizes markdown-safe context projection.  
   **Impact:** prompt builder + session manager depend on one formatter.

2. **File:** `packages/core/src/learnings.ts`  
   **Signatures:**  
   `export function readLearnings(projectPath: string): string | null;`  
   `export function mergeLearningEntry(projectPath: string, entry: { context: string; learning: string; rule: string; evidence: string; confidence: "low" | "medium" | "high"; lastValidated?: string }): { updated: boolean };`  
   **Rationale:** isolate `LEARNINGS.md` parsing/merge logic from prompt/session code.  
   **Impact:** reusable by spawn flow and completion reflection flow.

3. **File:** `packages/core/src/types.ts`  
   **Signatures:**  
   `export type WorkerRespawnStrategy = "resume" | "fresh" | "context-inject";`  
   Add to `ProjectConfig`: `workerRespawnStrategy?: WorkerRespawnStrategy;`  
   **Rationale:** required policy control for resume-first behavior.  
   **Impact:** config validation + session-manager selection logic.

4. **File:** `packages/cli/src/commands/context.ts`  
   **Signatures:**  
   `export async function exportContextCommand(args: { project?: string; output?: string }): Promise<void>;`  
   `export async function importContextCommand(args: { project?: string; input: string }): Promise<void>;`  
   **Rationale:** portability requirement in spec.  
   **Impact:** new CLI registration and docs.
