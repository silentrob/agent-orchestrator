## Context Summary

**Plan 0004** was intentionally **research and alignment**: specs (`issue-lifecycle-trust-vector.md`, `role-typed-artifacts.md`, errata, persistence sketch), reverted **0003**-style handoff as the north star, and shipped **vocabulary only** in core (`IssueWorkflowPhase`, `TrustGateKind`, `ISSUE_WORKFLOW_PHASES`, `TRUST_GATE_KINDS` in `issue-lifecycle-types.ts`). No runtime wiring.

**Plan 0005** executes the next layer: a **per-session (issue-backed) lifecycle** — store **phase** and **gate** state, inject **phase-appropriate** prompt layers, and surface **visibility** (dashboard / CLI / orchestrator) so operators and agents share one model of “where this issue is.”

**Depends on:** [`0004_PLAN.md`](./0004_PLAN.md) (approved), [`issue-lifecycle-trust-vector.md`](../docs/specs/issue-lifecycle-trust-vector.md), [`issue-lifecycle-state-persistence.md`](../docs/specs/issue-lifecycle-state-persistence.md).

---

## Problem

- Types exist but **nothing reads or writes** `issuePhase` / satisfied gates on a live session.
- **`buildPrompt`** does not accept **phase**; only **workerRole** at spawn is partially modeled (planner layer). Phase-aware stacking (planner vs executor vs validator instructions in **one** session) is **not** implemented.
- **Dashboard / `ao status`** do not show **current phase** or **next gate**, so humans cannot coordinate with the Trust Vector.
- **Policy** (advance only when gates pass) is **optional** in MVP but the hooks should exist (config or no-op).

---

## Approach (high level)

1. **Persistence:** Start with **session `metadata`** keys (see persistence sketch) for `issueWorkflowPhase` and optional serialized **satisfied gates** or boolean keys — minimal, no issue-index file until needed.
2. **Prompts:** Extend **`PromptBuildConfig`** with optional **`issueWorkflowPhase`** (or derive from metadata when building from session). Compose **one** additional block: pick `buildPlannerArtifactLayer` / future executor / validator layers by phase (same session, different phase = different L4.5).
3. **Visibility:** Expose phase (and later: next gate hint) on **session list/detail** API and **`ao status`** output when metadata present.
4. **Gates (later tasks in this plan or follow-up):** Orchestrator/lifecycle **hooks** to set phase when gates satisfied; **spawn guard** behind config flag — **optional** for first mergeable slice.

---

## Scope fence

### IN-SCOPE

- Canonical **metadata key names** and **read/write** from session manager / spawn defaults for issue-backed sessions.
- **`PromptBuildConfig` + `buildPrompt`** extension for phase-aware prompt injection (planner layer already exists; executor/validator layers **stub or thin** if not yet implemented).
- **Web + CLI** read path: phase visible where sessions are listed or detailed (minimal UI; Tailwind-only web).
- **Tests:** core prompt + metadata; web/API if routes touched.
- **Documentation:** short addition to `AGENTS.md` or spec pointer — only if needed for operator workflow.

### OUT-OF-SCOPE (initially)

- Full **artifact manifest** / `ao artifact publish`.
- **Guaranteed** enforcement of gates across all agents (best-effort + policy).
- Replacing **SSE interval** or large dashboard redesign.
- **Tracker** two-way sync of phase (labels) — optional follow-up.
